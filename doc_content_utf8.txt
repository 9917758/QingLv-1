二、需求分析
2.1 项目背景
随着人们健康意识的不断提升，个人健康数据的追踪与管理已成为日常生活中的重要需求。传统的健康管理方式存在数据分散、记录不便、缺乏专业指导等问题，用户难以全面了解自身健康状况并获得针对性的健康建议。
与此同时，鸿蒙操作系统（HarmonyOS）作为新一代智能终端操作系统，提供了分布式能力、原子化服务、多端协同等特性，为健康管理类应用带来了全新的发展机遇。基于鸿蒙生态开发健康管理应用，能够充分利用系统级能力，为用户提供更加便捷、智能的健康服务体验。
在此背景下，"清律"智能健康管理App应运而生，旨在基于HarmonyOS ArkTS技术栈，打造一款集健康数据追踪、AI健康咨询、健康知识科普、提醒打卡于一体的个人健康管理应用，帮助用户建立科学的健康管理习惯，提升整体健康水平。
2.2 项目目标
2.2.1 总体目标
开发一款基于HarmonyOS的个人健康管理应用，实现健康数据的全面追踪、智能分析与个性化指导，帮助用户建立健康的生活方式。
2.2.2 具体目标
1.
实现多维度健康数据的录入、存储与可视化展示，包括体重、BMI、体脂率、血压、心率、血糖等核心指标
2.
提供AI健康顾问功能，结合用户健康数据给出个性化健康建议
3.
构建健康知识库，提供专业的健康知识文章供用户学习参考
4.
建立健康提醒与打卡机制，帮助用户养成良好的健康习惯
5.
支持云端数据同步，实现多设备数据一致性与数据备份
6.
适配深色模式与多语言，满足不同用户的使用偏好
7.
支持桌面服务卡片，实现关键信息快速预览与快捷操作
8.
掌握鸿蒙移动应用的开发方法，熟悉ArkTS开发语言与HarmonyOS应用架构

5.1 前端技术实现
5.1.1 项目架构设计
本项目采用HarmonyOS多模块化架构，按照功能职责将代码划分为
：
commons（公共模块）、features（业务模块）、components（业务组件）和products（应用入口）四大层级。
commons模块封装了HTTP请求（HttpUtil）、JWT Token管理（AuthToken）、后端地址配置（NetworkConfig）、主题管理（ThemeUtil）和国际化（I18nUtil）等通用能力；features模块按业务领域划分为health（健康管理）、person（个人中心）和device（设备连接）三个子模块；components模块存放可复用的业务组件如数据仪表盘。这种分层架构实现了高内聚低耦合，各模块可独立开发、测试和维护。
5.1.2 状态管理方案
前端采用HarmonyOS 5.0推荐的@ObservedV2 + @Trace + PersistenceV2状态管理方案。
@ObservedV2装饰器标记可观察类，@Trace装饰器标记可追踪属性，当被追踪的属性发生变化时，UI组件自动触发重新渲染。@Computed装饰器用于派生计算属性，如主题颜色会随当前主题模式自动计算。PersistenceV2提供了本地数据持久化能力，所有健康数据、用户偏好均通过此机制实现本地存储，支持按用户ID进行数据隔离，确保多账号切换时数据互不影响。
5.1.3 路由与导航
项目采用自研OHRouter路由模块进行页面导航管理。OHRouter封装了HarmonyOS的Navigation和NavPathStack能力，提供了统一的页面跳转、参数传递和返回栈管理接口。关键改进包括：移除部分组件中不当的onPop回调（如deletePage()），避免双重弹栈导致的白屏崩溃问题；使用emitter事件机制协调跨页面状态更新（如个人信息修改后通知首页刷新）。这些改进有效解决了导航相关的稳定性问题。
5.1.4 Mock数据与开发环境
为支持纯App环境的独立开发和测试，项目实现了完整的Mock数据体系。StepMockData模块采用动态日期生成算法，根据当前日期自动生成周/月/三个月等不同时间跨度的模拟步数数据，避免了硬编码日期导致的未来日期问题。所有Mock数据模块（血糖、血压、睡眠、周报、月报等）均在USE_MOCK_DATA标志位控制下运行，确保数据逻辑与统计值的一致性。模拟数据包含合理的随机波动和长期趋势，为前端功能验证提供了可靠的数据基础。
5.1.5 Markdown渲染
AI助手回复中的Markdown格式文本需要在ArkTS环境中正确渲染。由于ArkTS的@Builder函数不支持for循环和let变量声明，项目采用了预处理方案：在前端使用parseMarkdown()函数将Markdown文本解析为MarkdownSegment数组（每个segment包含text和bold属性），然后将segments存储在DisplayMessage接口中。渲染时，使用Text组件嵌套Span组件的方案，根据每个segment的bold属性设置Span的fontWeight样式，实现了加粗文本的正确显示。对于列表、标题等其他Markdown语法，通过cleanMarkdownText()函数进行清理处理。
5.1.6 深色模式与国际化
ThemeUtil工具类实现了浅色/深色/跟随系统三种主题模式的管理。通过@Computed计算属性，所有页面的主题颜色随当前主题模式自动计算和更新，主题切换即时生效。I18nUtil工具类实现了跟随系统语言的自动切换，所有11个模块均配置了中英文双语资源文件（element/string.json和en_US/element/string.json），页面中的硬编码字符串已替换为$r('app.string.xxx')资源引用。
5.2 后端技术实现
5.2.1 分层架构
后端采用FastAPI + SQLAlchemy + MySQL技术栈，遵循分层架构设计：路由层（routers/）负责HTTP请求接收和响应，定义了50+个RESTful API端点；服务层（services/）封装核心业务逻辑，包括用户认证、健康数据管理、AI分析等；数据模型层（models/）使用SQLAlchemy ORM定义数据库表结构，包含User、HealthRecord、WaterIntake、ReminderPlan等10个数据模型；数据校验层（schemas/）使用Pydantic v2进行请求/响应的字段级校验，所有数值字段均设置了合理的上下界约束。
5.2.2 认证与安全
用户认证采用JWT Bearer Token方案。用户注册时密码经bcrypt算法加密后存储，登录成功后服务端生成JWT Token返回客户端。AuthToken工具类负责Token的本地持久化和自动过期处理。后端通过JWT Bearer认证中间件保护所有需登录的API接口，每次请求验证Token有效性。所有本地数据按用户ID隔离存储，切换账号不会泄露其他用户数据。此外，后端实施了输入数据严格校验、SQL注入防护（ORM参数化查询）和异常信息脱敏等安全措施。
5.2.3 数据完整性保障
数据库层面通过唯一约束保障关键数据的完整性：water_intake表的(user_id, date)唯一约束确保每人每天一条饮水记录；checkin_records表的(user_id, plan_id, checkin_date)唯一约束防止重复签到；sleep_records表的(user_id, date)唯一约束保证每人每天一条睡眠记录；chat_messages表的外键ON DELETE CASCADE确保删除会话时自动清理关联消息。业务层面，饮水打卡使用UPDATE SET cup_count = cup_count + 1原子操作避免并发竞争；签到操作通过IntegrityError捕获兜底防止唯一约束冲突；并发请求通过IntegrityError捕获和db.rollback()异常回滚机制保障数据一致性。
5.2.4 API接口设计
后端共设计实现了50+个RESTful API接口，覆盖用户认证（注册/登录/个人信息）、健康数据CRUD、仪表盘与趋势分析、饮水管理、提醒计划与打卡、体重目标管理、睡眠管理、健康知识浏览与收藏、AI分析与会话管理等全部业务场景。所有接口遵循RESTful规范，使用标准的HTTP方法（GET/POST/PUT/DELETE）和状态码。请求体与响应体均使用Pydantic v2 Schema进行严格校验，启动后自动生成Swagger API文档（/docs端点），便于前端开发时的接口调试与对接。
5.3 AI Agent技术实现
5.3.1 ReAct Agent架构
AI健康顾问是本项目的核心技术亮点，采用ReAct（Reasoning + Acting）Agent架构实现。与传统的单轮LLM对话不同，ReAct Agent能够自主决策：当用户提问时，LLM首先进行推理，判断是否需要调用工具获取数据；如需调用，则生成工具调用指令；后端执行工具后，将结果反馈给LLM继续推理；此过程可循环多轮（最多6轮），直到LLM获得足够信息给出最终回答。这种架构使AI能够动态获取用户真实健康数据，而非仅依赖训练时的通用知识。
5.3.2 工具系统设计
Agent工具系统定义了10个专业工具，覆盖健康管理的各个维度：get_user_profile（查询用户个人信息）、get_health_records（查询历史健康数据）、get_today_water_intake（查询今日饮水）、get_weight_goal（查询体重目标）、get_checkin_streak（查询打卡记录）、get_favorite_articles（查询收藏文章）、analyze_nutrition（食物营养分析，内置食物营养数据库）、generate_exercise_plan（生成运动计划，按减重/增肌/保持不同目标模板化生成）、predict_health_trend（健康趋势预测，基于历史数据线性回归）和assess_health_risk（综合风险评估）。工具函数采用闭包模式绑定db session和user_id，确保数据隔离和安全访问。
5.3.3 技术选型与演进
AI模块最初考虑使用LangGraph的create_react_agent预制方案，但在实际集成过程中遇到了langgraph-prebuilt与langchain-core版本的兼容性冲突（TOOL_MESSAGE_BLOCK_TYPES导入错误）。经分析后，团队决定放弃langgraph依赖，转而使用LangChain原生的llm.bind_tools()方法手动实现ReAct Agent循环。手动实现方案更简洁、依赖更少、调试更灵活，且不依赖特定框架版本。Agent循环的核心逻辑仅约40行代码，易于理解和维护。此外，LLM异常时实现了三层回退机制：Agent异常→简单对话模式→规则引擎，保障服务的高可用性。
5.3.4 对话上下文管理
每次AI对话前，后端自动构建丰富的上下文信息：查询用户的个人信息（性别、身高、体重、年龄）、近7天健康数据（体重、血压、心率、血糖等）、体重目标与进度、今日饮水情况和收藏文章列表。这些上下文以SystemMessage形式传入LLM，使AI无需每次手动查询即可了解用户基本情况。对话历史支持多轮记忆，最多保留最近50条消息（通过Pydantic Schema的max_length=50限制），确保长对话不超出LLM的上下文窗口限制。
5.4 关键技术难点与解决
1.
导航栈白屏问题：部分组件在onPop回调中调用了deletePage()，导致Navigation导航栈被双重弹出，页面显示白屏。通过仔细分析ArkUI的导航生命周期，确认onPop回调执行时页面已被系统自动弹出，不需要手动deletePage。解决方案是移除所有组件中不当的onPop: deletePage()回调，涉及HealthRunningRow、HealthRunningFoldRow、HealthCardRow等多个组件。
2.
ArkTS Builder函数限制：AI助手的Markdown渲染需要在@Builder函数中进行条件判断和循环处理，但ArkTS的@Builder函数有严格的语法限制（不支持for循环、let变量声明）。解决方案是将Markdown解析逻辑前置到消息创建阶段，使用parseMarkdown()函数预解析为结构化的MarkdownSegment数组，渲染时仅做简单的条件判断（if/else），绕过了@Builder的限制。
3.
后端依赖版本冲突：langgraph.prebuilt依赖的TOOL_MESSAGE_BLOCK_TYPES符号在所安装的langchain-core版本中不存在，导致ImportError。深入分析后发现langgraph与langchain-core版本不兼容。解决方案是手动实现ReAct Agent循环，完全移除langgraph依赖，仅使用LangChain的bind_tools()原生能力，既解决了兼容性问题又简化了项目依赖。
4.
AI体重数据缺失：用户反映AI知道身高（173cm）但不知道体重。经排查发现，后端查询用户信息时只从HealthRecord表读取健康数据，而身高存储在User表中、体重在User表中有值但HealthRecord中可能为空。解决方案是在agent_chat()和chat_with_advisor()中添加从User表读取体重信息并纳入上下文的逻辑。
5.
体重单位统一：项目早期存在体重单位不统一的问题，前端部分页面使用斤、后端使用公斤。为保证数据一致性，项目中进行了全面的单位统一工作：前端所有体重显示和录入统一为公斤，后端所有体重存储和计算统一为公斤，移除了所有斤/公斤转换逻辑（如/2运算），数据库表中的体重字段也统一为公斤单位。

6.1 项目成果
经过团队成员的共同努力，清律智能健康管理App已完成全部核心功能的开发与测试。项目实现了健康数据管理（支持体重、BMI、体脂率、血压、心率、血糖、步数、睡眠等8种指标的录入、展示与趋势分析）、AI健康顾问（基于ReAct Agent架构，支持10个智能工具的多轮对话）、健康知识库（30篇专业文章，支持分类筛选与收藏）、提醒打卡系统（饮水/用药提醒、连续打卡统计、成就徽章）、云端数据同步（基于JWT认证的50+个RESTful API）、深色模式与多语言支持、桌面服务卡片等七大功能模块。
前端代码采用模块化架构，共实现页面30+个、组件40+个、服务类20+个，代码总量超过20,000行。后端实现API接口50+个、数据模型10个、Pydantic Schema 30+个，代码总量约3,500行。前后端均已通过完整的联调测试，所有功能正常运行。项目已参与华为鸿蒙应用激励计划，提交了完整的应用作品和项目文档。
6.2 技术收获
1.
深入掌握了HarmonyOS ArkTS开发：通过本项目，团队成员熟练掌握了ArkTS语言语法、@ComponentV2/@ObservedV2/@Trace等状态管理机制、Navigation导航栈管理、@Builder自定义构建函数、Form Ability桌面卡片开发等核心能力，积累了丰富的鸿蒙应用开发经验。
2.
理解了ReAct Agent架构原理：从最初的简单LLM对话模式，到LangGraph预制方案，再到手动实现ReAct Agent循环，团队逐步深入理解了AI Agent的工作机制。手动实现过程中，深刻理解了tool calling（bind_tools）、消息流管理（SystemMessage、HumanMessage、AIMessage、ToolMessage的传递顺序）和Agent循环控制（最大轮次、异常回退）等核心概念。10个工具的设计与实现使团队理解了如何将业务能力封装为LLM可调用的结构化工具。
3.
掌握了全栈健康管理系统的设计方法：从需求分析到技术选型，从前端UI设计到后端API设计，从数据模型设计到数据完整性保障，团队完成了一个完整的全栈项目的开发全流程。特别是在健康数据管理领域，学习了医学指标的合理范围校验、数据趋势分析、异常检测等专业知识。
4.
提升了工程化开发能力：项目中实践了版本控制（Git分支管理）、文档编写（README、CHANGELOG、API文档、项目报告）、错误处理策略（三层回退、异常日志记录）、依赖管理（版本兼容性处理）、并发安全（原子操作、唯一约束）等工程化实践。
6.3 主要挑战与反思
1.
HarmonyOS生态成熟度：在开发过程中遇到了ArkTS语法限制严格、部分API文档不够详尽、DevEco Studio调试工具有待完善等问题。这要求团队投入额外的时间进行摸索和试错，但也加深了对鸿蒙系统底层原理的理解。随着鸿蒙生态的持续发展，此类问题有望逐步改善。
2.
AI能力与用户体验的平衡：AI Agent虽然功能强大，但LLM的响应速度受限于网络和模型推理时间，用户在等待AI回答时可能感到延迟。项目通过设置超时机制、回退策略和加载状态提示来改善体验，但AI响应延迟仍是在线AI应用的共性挑战。未来可考虑引入流式输出（Streaming）和本地小模型加速。
3.
体重单位统一的历史债务：项目早期未严格统一单位为公斤，导致部分页面存在斤/公斤混用的情况。虽然最终完成了全面排查和修正，但这一经历表明在数据密集型应用中，及早建立并严格执行数据规范（特别是单位、时区、货币等）至关重要。
6.4 未来展望
1.
流式对话：引入Server-Sent Events（SSE）或WebSocket实现AI回答的流式输出，减少用户等待感，提升对话体验。
2.
数据可视化增强：引入更丰富的图表类型（如雷达图、热力图），增强健康数据的可视化表现力。
3.
设备直连健康数据：接入华为Health Kit或蓝牙健康设备，实现血压计、血糖仪等设备的自动数据采集，减少用户手动录入的负担。
4.
社区功能：增加用户社区模块，支持健康目标分享、运动打卡排行、健康知识讨论等社交功能，提升用户参与度和留存率。
5.
AI Agent能力扩展：增加更多专业工具（如用药建议查询、体检报告解读、中医体质辨识等），以及支持多模态输入（如拍照识别食物
计算热量）。
